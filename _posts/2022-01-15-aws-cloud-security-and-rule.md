## <span style="color:#0f7b6c">1. AWS 클라우드 보안</span>

#### 1-1. AWS 공동 책임 모델

AWS 공동 책임 모델이란 AWS 클라우드 서비스를 사용하는데 있어서 보안과 규정 준수가 AWS와 고객의 공동 책임이라는 의미다. 그리고 AWS가 책임을 져야할 부분과 고객이 책임을 져야할 부분이 명시적으로 나뉜다.  

![aws-shared-responsibility-model]({{site.url}}/img/AWS/aws-shared-responsibility-model.jpg){:width="800"}

<div style="text-align:center">이미지 출처 : https://aws.amazon.com/ko/compliance/shared-responsibility-model/</div>

위 사진에 따르면 **클라우드의 보안**은 AWS의 책임이다. 즉, AWS는 물리적 시스템, 시설과 관련된 모든 인프라 자체를 책임을 지겠다는 의미다. 그리고 고객은 **클라우드에서의 보안**을 책임진다. 클라우드 내에서 구현하는 모든 애플리케이션과 데이터는 고객이 보호할 책임을 갖는다는 의미다. AWS가 제공하는 보안, 규정 도구를 이용해서 말이다. 시스템간 전송중인 데이터의 암호화도 해당한다. 때문에 고객은 네트워크가 보안을 보장하도록 구성되어 있는지 확인해야 한다.

지금부터 알아볼 보안 및 규정은 주로 **클라우드에서의 보안**을 위한 내용이다. AWS가 책임지지 않고 고객이 책임질 의무가 있는 보안, 규정을 잘 관리하기 위한 도구와 서비스에 대해 알아볼 것이다.

#### 1-2. 클라우드 서비스 특성에 따른 고객의 보안 책임

클라우드에서의 보안 또한 AWS가 제공하는 클라우드 서비스의 특성과 함께 다를 수 있다. 

- IaaS(서비스형 인프라):
    + 고객은 네트워킹 및 스토리지 설정을 보다 유연하게 할 수 있음
    + 보안의 더 많은 측면을 고객이 관리해야함
    + 엑세스 제어를 고객이 구성해야함
    + EC2, EBS, VPC
- PaaS(서비스형 플랫폼):
    + 고객이 기본 인프라를 관리할 필요가 없음
    + 운영 체제, 데이터베이스 패치 및 적용, 방화벽 구성 및 재해 복구를 AWS가 책임
    + 고객은 코드 또는 데이터 관리에 집중
    + Lambda, RDS, Elastic Beanstalk
- SaaS(서비스형 소프트웨어):
    + 소프트웨어가 중앙에서 호스팅됨
    + 구독 모델 또는 과금제로 라이선스가 부여됨
    + 일반적으로 웹 브라우저, 모바일 앱 또는 API를 통해 서비스에 엑세스
    + 고객은 서비스를 지원하는 인프라를 관리할 필요가 없음
    + AWS Trusted Advisor

#### 1-3. AWS IAM(AWS Identify and Access Management)

고객은 IAM 서비스를 사용하여 AWS 리소스에 대한 엑세스를 관리할 수 있다. IAM 서비스를 이해하기 위해서는 AWS의 특별한 사용자 서비스를 이해할 필요가 있다. 가장 처음 AWS 계정을 생성하고 로그인을 하면 우리는 루트 사용자로 로그인을 하게 된다. 이러한 루트 사용자는 쉽게 말해 하나의 계정 내 모든 권한을 갖고 있는 사용자를 의미한다. MS Windows OS로 비교하자면 관리자 계정으로 윈도우에 로그인 할 수 있고 그 외에 권한이 제한된 손님 계정 등을 만드는 식이라고 볼 수 있다. 

모든 권한을 갖는 사용자를 사용하는 것은 서비스 운영에 있어 위험이 따른다. 따라서 고객은 데이터 베이스를 관리하는 사용자, 서버를 관리하는 사용자 등으로 하나의 계정에 여러 사용자를 만들고 싶을 수 있다. 이때 사용하는 개념이 바로 IAM이다. 고객은 IAM 서비스를 통해 사용자를 생성하고 각각의 사용자에게 특정 권한을 부여(루트 사용자 환경에서) 할 수 있다. 그리고 이때 부여한 권한이 각각의 IT 리소스에 대한 접근(엑세스), 인증, 관리 등의 권한에 해당한다.  

AWS ORGANIZATIONS은 통합 결제, 여러 AWS 계정을 관리 등을 위해 서로 다른 여러 AWS 계정을 묶는 개념이라면, IAM은 하나의 계정 내에서 여러 사용자를 만들고 각각의 권한을 다르게 부여하는 개념이다. 

다음 사진은 AWS Academy 과정에서 부여받은 계정으로 결제 서비스에 접근한 사진이다. 사진을 보면 권한이 없는 것을 볼 수 있는데 해당 계정이 특정 권한만 사용하도록 부여된 IAM 계정임을 알 수 있다.

![aws iam]({{site.url}}/img/AWS/aws-iam.png){:width="800"}

기본 적으로 루트 계정을 제외하고 생성된 사용자는 모든 권한이 거부된 상태이다. 고객은 필요에 맞게 권한을 할당할 필요가 있으며 당연하게 최소한의 권한을 부여(최소 권한 원칙) 하는게 이상적이다. 만약 특정 리소스에 대한 권한이 한번이도 거부된다면 추후 해당 리소스에 대한 허용 권한이 나오더라도 해당 권한은 거부된다. 즉, 거부 권한과 허용 권한이 상충되면 거부 권한이 우선시 된다는 이야기다. 

#### 1-4. 새 AWS 계정에 대한 보안

앞서 이야기했듯 루트 사용자를 사용하는 것은 위험하다. 때문에 필요한 경우를 제외하고 AWS 계정 루트 사용자를 사용하는 것은 바람직하지 않다. 

**# 루트 사용자만 수행할 수 있는 작업의 예**

- 계정 루트 사용자 암호 업데이트
- AWS Support Plan 변경
- IAM 사용자의 권한 복원
- 계정 설정 변경(연락처, 허용 리전 ...)

위와 같이 루트 사용자는 필요할 때만 사용하고, 새로운 AWS 계정을 생성하면 가능한 빨리 루트 사용자 계정의 사용을 중지해 새로운 계정에 대한 보안을 강화하는게 바람직하다. 

**# AWS Cloud Trail 사용**

Cloud Trail은 계정의 사용자 활동을 추적하는 서비스다. 계정에서 지원되는 모든 서비스의 리소스에 대한 모든 API 호출을 로깅한다. 기본적으로 AWS Cloud Trail 이벤트 기록은 활성화되어 있으며 무료이다. 기본 활성화된 추적의 경우 최대 90일간의 계정 활동에 대한 모든 관리 이벤트 데이터가 포함된다. 만약 90일이 지난 로그를 홯성화하고 지정된 이벤트 알림 기능을 추적하려면 새로운 추적을 생성해야한다. 

#### 1-5. 계정 보안

##### 1-5-1. AWS Organizations

앞서 AWS 결제 관련 서비스를 알아보며 통합 결제를 위한 방법으로 AWS Organizations을 살펴봤다. 여기서는 좀 더 보안기능에 초점을 맞춰 해당 서비스를 살펴보면, AWS Organizations는 SCP(Service Control Policy)라는 엑세스 정책을 갖는다. 해당 정책을 특정 계정이나 계정 그룹(OU)에 할당할 수 있으며, 이는 IAM이 동작하는 방식과 비슷하게 특정 계정의 AWS 리소스 접근을 결정한다. IAM은 계정 내 사용자(또는 사용자 그룹)에 대한 AWS 리소스 접근을 관리한다면, AWS Organizations은 계정(또는 계정 그룹) 자체의 AWS 리소스 권한을 관리한다고 볼 수 있다. 

##### 1-5-2. AWS KMS(AWS Key Management Service)

- 암호화 키를 생성하고 관리할 수 있다.
- AWS 서비스 및 애플리케이션 전체의 암호화 사용을 제어할 수 있다.
- AWS Cloud Trail과 통합하는 경우 모든 키 사용을 로깅한다.
- FIPS(Federal Information Processing Standards) 140-2에서 검증한 HSM(하드웨어 보안 모듈)을 사용하여 키를 보호한다.

##### 1-5-3. Amazon Cognito 기능

- 웹 및 모바일 애플리케이션에 사용자 가입, 로그인 및 엑세스 기능을 추가할 수 있다.
- 수백만 명의 사용자까지 확장된다.
- Fackbook, Google 및 Amazon과 같은 소셜 증명 공급자와 SAML(Security Assertion Markup Language) 2.0을 사용한 Microsoft Active Diretory와 같은 엔터프라이즈 자격 증명 공급자를 사용한 로그인을 지원한다.

##### 1-5-4. AWS Shield 기능

- DDoS(분산 서비스 거부 공격) 방어를 위한 관리형 서비스
- AWS에서 실행되는 애플리케이션을 보호한다.
- 상시 탐지 및 자동 인라인 완화 제공한다.

해당 서비스를 통해 애플리케이션 다운타임과 지연 시간을 최소화 할 수 있다.

#### 1-6. AWS의 데이터 보안

AWS는 유휴 데이터(디스크 또는 테이프에 물리적으로 저장된 데이터)의 암호화를 지원한다. 암호화는 읽을 수 없는 보안 키를 이용해 데이터를 인코딩하며, 보안 키를 가지고 있는 사용자만 데이터를 디코딩해 읽을 수 있다. 그리고 이러한 보안 키는 AWS KMS에서 관리할 수 있다. Amazon S3, EBS, EFS, RDS를 포함하여 KMS가 지원하는 모든 서비스에 저장된 데이터를 암호화 할 수 있다.

##### 1-6-1. 전송 중 데이터의 암호화

전송 중 데이터란 네트워크에서 이동하는 데이터를 나타낸다. 전송 중 데이터는 전송 계층 보안(TLS 1.2)를 사용하여 암호화 된다. AWS Cetificate Manager는 TLS 또는 SSL 인증서 관리, 배포 및 갱신하는 방법을 제공한다. 

##### 1-6-2. Amazon S3 버킷 및 객체 보안

- 새로 생성된 S3 버킷은 기본적으로 비공개이며 보호된다.
- Amazon S3와 데이터 객체를 공유해야 하는 사용사례의 경우:
    + 데이터 엑세스를 관리하고 제어하는 것이 필수
    + 최소 권한 원칙을 준수하는 권한을 따르고, Amazon S3 암호화 사용을 고려해야 한다.
- S3 데이터 엑세스 제어를 위한 도구 및 옵션:
    + Amazon S3 Block Public Access 기능(사용이 간단)
    + IAM 정책(사용자가 IAM을 사용해 인증할 수 있는 경우 좋은 옵션)
    + 버킷 정책
    + ACL(엑세스 제어 목록)
    + AWS Trusted Advisor 버킷 권한 확인

## <span style="color:#0f7b6c">2. AWS 규정 준수</span>

AWS를 이용하는 고객이 매우 다양한 만큼 고객이 하고있는 비즈니스도 다양하다. 이에 따라 각각의 비즈니스에 적용되는 보안과 준수해야 할 규정 또한 매우 다양해질 수 있다. 간단한 예로 유럽인을 대상으로 비즈니스를 한다면 GDPR(유럽연합 일반 데이터 보호 규칙)에 따라야 할 것이다. 이렇게 다양한 규정을 준수해야 하는 고객을 위해 AWS가 제공하는 프로그램이 바로 AWS 규정 준수이다. 만약 온프레미스 환경을 이용한다면 매우 까다로워질 수 있는 보안 및 규정준수를 AWS 서비스를 이용해 더욱 쉽게 보장할 수 있는 것이다. 

규정 준수 프로그램은 넓은 범위로 다음과 같이 분류될 수 있다.

- 인증 및 인가:
    + 독립적인 외부 감사자가 평가
    + 예: ISO 270001, 27017, 27018 및 ISO/IEC 9001
- 법률, 규정 및 개인 정보:
    + AWS는 보안 기능 및 법적 계약을 제공하여 규정 준수를 지원
- 준수 및 프레임워크:
    + 산업별 또는 기능별 보안 또는 규정 준수 요구 사항
    + 예: CIS(Center for Internet Security), EU-미국 프라이버시 실드 인증

#### 2-1. AWS Config

비즈니스에 따라 규정이 다르며 하나의 비즈니스에 존재하는 여러 부서/프로젝트에 따라서도 규정이 달라진다. 그 때문에 기존에 규정을 관리하기 위해서는 각 부서에 규정 준수가 제대로 이뤄지고 있는지 확인 요청을 보내고, 각각의 부서는 요청에 따라 규정 준수가 제대로 이뤄지는지 확인해야 했다. 이러한 과정은 매우 번거롭고, 비용이 많이 든다. 이러한 번거로운 과정을 해결하기 위한 AWS 서비스가 AWS Config다. 

AWS Config는 내부적으로 Inventory라는 개념을 갖는다. Inventory는 관리 대상이 되는 AWS 리소스 목록과 해당 리소스의 규정 준수 여부에 대한 데이터를 갖는다. 또한 Inventory는 리소스 구성에 대한 변경 사항, 설정 등도 수집한다. 

- AWS 리소스의 구성을 측정, 감사 및 평가
- 구성을 지속적으로 모니터링하는데 사용
- 기록된 구성과 원하는 구성을 자동으로 비교하여 평가
- 구성 변경을 검토
- 세부 구성 기록을 본다.
- 규정 준수 감사 밒 보안 분석을 간소화

#### 2-2. AWS Artifact

- 규정 준수 관련 정보를 위한 AWS 리소스 (온디맨드 다운로드 제공)
- 보안 및 규정 준수 보고서에 엑세스하고 온라인 계약을 선택할 수 있음
- 예제 다운로드에 엑세스할 수 있음
    + AWS ISO 인증
    + PCI(Payment Card Industry) 및 SOC(Service Organization Control) 보고서
- AWS Management Console에서 직접 AWS Artifact에 엑세스할 수 있음
